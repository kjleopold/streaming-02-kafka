## Custom Project

### Dataset

The dataset used by the Kafka producer was sales.csv. The dataset contains
simulated online course sales from customers in different regions.

Each record includes fields such as:

- order_id
- datetime
- region_id
- currency_code
- product_id
- unit_price
- quantity
- is_online
- customer_id
- is_new_customer
- device_type
- payment_method
- referral_source
- discount_code
- customer_note

I used the original dataset provided for the project, but I modified the
streaming messages before sending them through Kafka by adding additional
calculated fields.

### Kafka Messages

The Kafka producer streamed enriched sales transaction messages through Kafka
one message at a time.

The Kafka topic used for the project was streaming-02-kafka-kjleopold.

The message key used was the region_id field. This grouped messages by region
inside Kafka partitions.

I modified the producer to add two calculated fields to each message titled
total_revenue and sales_tier.

### Consumer Processing

The Kafka consumer received the enriched sales transaction messages from the
Kafka topic.

During testing, the consumer processed up to 10 messages and read them from the
beginning of the topic.

For each message, the consumer:

- logged the message contents
- processed the Kafka event data
- appended the records to an output CSV file

The processed records were written to consumed_sales_kjleopold.csv.

The consumer did not filter out any records, but it successfully processed and
stored all of the enriched streaming events.

### Experiments

#### Phase 4

I changed the producer configuration to increase the number of
messages from 3 to 10 by updating the PRODUCER_MESSAGE_COUNT setting in the
.env file.

I also changed the message interval from 2.0 seconds to 1.0 second using the
PRODUCER_MESSAGE_INTERVAL_SECONDS setting. This made the stream move faster
and allowed more messages to flow through Kafka during testing.

Lastly, I created and used a custom Kafka topic named streaming-02-kafka-kjleopold.

#### Phase 5

I modified the Kafka producer to enrich the sales event data before
publishing the messages to Kafka.

Two new fields to each message titled total_revenue and sales_tier.

The total_revenue field was calculated by multiplying the unit price by the
quantity purchased.

The sales_tier field classified transactions as BASIC, STANDARD, or PREMIUM
based on the total revenue amount.

### Results

When I ran the producer and consumer, the producer successfully streamed the
enriched sales transaction messages into Kafka, and the consumer successfully
received and processed them.

The output CSV confirmed that all messages were transferred correctly,
calculated fields were preserved, Kafka partition and offset metadata were
recorded, and the consumer received enriched event data exactly as produced.

The test showed that the producer and consumer were able to reliably send and
receive enriched messages through Kafka.

### Interpretation

This project helped me better understand how Kafka producers and consumers work
together to process data in motion.

Compared to the original example, I modified the producer so it performed event
enrichment before sending messages through Kafka. Instead of sending only the
raw CSV records, the producer generated more meaningful events that included
calculated revenue and sales tier information.

Watching the messages move through Kafka made it easier to understand how
streaming systems continuously process and transport live event data between
applications in real time.

The stream could help a business identify higher-value transactions, prioritize
important orders, and monitor customer activity as events happen.

The consumed messages also provided useful business intelligence by transforming
raw transaction data into enriched event data that could later be used for
dashboards, reporting, monitoring, or operational decision-making.
